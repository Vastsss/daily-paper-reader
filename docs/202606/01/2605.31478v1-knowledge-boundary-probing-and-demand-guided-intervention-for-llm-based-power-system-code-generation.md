## 摘要
大型语言模型（LLM）正越来越多地被用于自动化电力系统分析，但许多公用事业公司和能源研究实验室出于保密、监管、可复现性和成本原因，需要本地部署服务。这使得开放权重模型的可靠性成为一个部署问题。我们展示了电力系统代码生成中的首次失败并非仅由推理主导，而是由结构化的API知识边界错误主导：在版本化的仿真库中出现虚构的函数名称、误用的参数以及错误处理的结果表。我们引入了PowerCodeBench，这是一个经过执行验证的基准生成器，它将自然语言操作员查询与pandapower代码和数值真实值配对；一个L0-L3文档驱动的探测程序，用于测量每个模型的API知识概况；以及一种边界感知干预，将查询端API需求估计与有针对性的主动文档注入和路由的反应性纠正相结合。在2000个任务的冻结版本上，我们评估了十个开放权重LLM（1.5B-480B参数）和四个商业中端API。该干预措施使每个评估的至少7B参数的开放权重模型和每个商业API的准确率提高了32到56个点。70B-120B范围内的开放权重模型与商业中端准确率范围相匹配，而Llama-3.1-405B和Qwen3-Coder-480B领先。有针对性的提示保留了全上下文准确率上限，同时使用了41%的提示令牌成本。结果是为电网分析工作流程提供了一条准确率侧、部署时间的路径，以实现可靠的本地LLM辅助，而无需微调或云推理。

## Abstract
Large language models (LLMs) are increasingly used to automate power-system analysis, but many utilities and energy-research labs require on-premise serving for confidentiality, regulatory, reproducibility, and cost reasons. This makes the reliability of open-weight models a deployment issue. We show that first-pass failures in power-system code generation are dominated not by reasoning alone, but by structured API-knowledge boundary errors: hallucinated function names, misused parameters, and mishandled result tables in versioned simulation libraries.   We introduce PowerCodeBench, an execution-validated benchmark generator that pairs natural-language operator queries with pandapower code and numerical ground truth; an L0-L3 documentation-driven probing procedure that measures per-model API knowledge profiles; and a boundary-aware intervention that combines query-side API demand estimation with targeted proactive documentation injection and routed reactive correction.   On a 2,000-task frozen release, we evaluate ten open-weight LLMs (1.5B-480B parameters) and four commercial mid-tier APIs. The intervention improves every evaluated open-weight model of at least 7B parameters and every commercial API by 32 to 56 accuracy points. Open-weight models in the 70B-120B range match the commercial mid-tier accuracy range, while Llama-3.1-405B and Qwen3-Coder-480B lead the panel. The targeted prompts preserve the full-context accuracy ceiling while using 41% of the prompt-token cost. The result is an accuracy-side, deployment-time path toward reliable on-premise LLM assistance for grid-analysis workflows without fine-tuning or cloud inference.

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- **诊断+干预**：先通过L0-L3分层探测（识别、回忆、理解、应用）量化每个模型对每个API的可靠性（风险得分），再根据查询的实际API需求，在生成前主动注入所需文档，并在执行失败后路由到针对性的修复提示。

### 关键技术细节
1. **L0-L3知识边界探测**：
   - L0（识别）：四选一测试，区分真实API名与虚构名。
   - L1（回忆）：要求生成函数签名和必要参数，用Jaccard相似度评分。
   - L2（理解）：多项选择，测试对函数用途、参数含义、返回信息的理解。
   - L3（应用）：要求写可执行代码并实际运行，以执行成功与否评分。
   - 每个函数f的**风险得分**：\( \rho_M(f) = 1 - \frac{1}{4}\sum_{l=0}^3 s_M^{(l)}(f) \)，其中\( s_M^{(l)}(f) \)为层得分。
2. **查询端需求估计**：
   - 训练一个**混合TF-IDF**分类器，预测自然语言查询需要哪些API函数。训练数据来自API遍历生成的6,312个(query, function-list)对。
   - 通过**角色频率重加权**适应目标查询分布（如PowerCodeBench查询），无需函数级标签，使用无标注目标语料计算角色比例，然后重训练分类器。
3. **边界感知干预**：
   - **主动阶段（第0轮）**：基于注入得分\( I_\ell(f,q,M) \)选择文档片段。公式：\( I_\ell = \hat{d}(f,q) \cdot \max(r_{M,\ell}(f), \tau_\ell(f,q)) \cdot a_\ell(q) \cdot w_\ell \)，其中\(\hat{d}\)是需求概率，\(r_{M,\ell}\)是模型在该层的缺陷，\(\tau_\ell\)是确定性底线，\(a_\ell\)是查询任务深度乘数，\(w_\ell\)是层权重。贪心选择高价值片段，限定token预算。
   - **反应阶段（第1-3轮）**：根据运行时异常类或值错误，路由器选择修复策略：
     - **代码错误路由**：基础修复（无文档）、API合同（注入L2调用合同）、边界合同（DataFrame模式提示）。
     - **值错误路由**：数值不匹配分支（提取AST执行轨迹，注入函数输出合同）和输出格式分支（修复打印格式）。

- **最小模型容量阈值**：1.5B模型即使使用完整方法准确率仍低于10%，表明方法对极小模型无效。实际部署建议≥7B。
- **封闭API覆盖有限**：仅测试4个供应商的标准模式，推理模式只在2个API的200任务子集上检查。
- **传递性适应假设**：需求估计的角色频率重加权需要一个**无标注的目标查询语料**（如PowerCodeBench查询），这是一种温和的传递性假设。如果没有这样的语料，性能会下降约10%（表A.2）。
- **未报告硬件指标**：未测量总拥有成本、延迟、吞吐量、能耗等，这些是实际部署的重要考量。
- **极端情况**：短期、状态估计等稀有API任务准确率仍较低（即使干预后），高要求场景仍需更大模型。
