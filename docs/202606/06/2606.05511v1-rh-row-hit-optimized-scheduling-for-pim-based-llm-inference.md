## 摘要
在处理中存储（PIM）架构上运行的大语言模型推理有望通过在HBM3 DRAM存储体内部直接执行乘累加（MAC）操作来突破内存墙。先前的工作将功率限制时序参数nCCDAB识别为主要的性能瓶颈，并据此优化调度。我们证明，对于主导自回归解码的GEMV操作，DRAM行周期时间（nRC）比nCCDAB大10到11倍。因此，nCCDAB完全被掩盖，使得先前专注于nCCDAB的优化对这些工作负载无效。根本原因在于继承的以主机为中心的地址交错，这迫使每个全存储体MAC命令进入不同的DRAM行。我们提出了RH+调度，一个简单的步长变化，使得连续的32个MAC操作保持在同一个行内。在四个LLM工作负载上的周期精确模拟显示，RH+实现了8-12倍的加速，超过74%的能耗降低，以及高达52倍的EDP改进。

## Abstract
Large language model inference on processing-in-memory (PIM) architectures promises to break the memory wall by performing multiply-accumulate (MAC) operations directly within HBM3 DRAM banks. Prior work identifies the power constraint timing parameter nCCDAB as the primary performance bottleneck and optimizes scheduling accordingly. We demonstrate that for GEMV operations that dominate autoregressive decoding, the DRAM row cycle time (nRC) is 10 to 11 times larger than nCCDAB. Consequently, nCCDAB is entirely masked, rendering prior nCCDAB-focused optimizations ineffective for these workloads. The root cause is inherited host-centric address interleaving, which forces every all-bank MAC command into a different DRAM row. We propose RH+ scheduling, a simple stride change that keeps 32 consecutive MAC operations within the same row. Cycle-accurate simulation across four LLM workloads shows that RH+ delivers 8-12x speedup, over 74% energy reduction, and up to 52x EDP improvement.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：大语言模型（LLM）在自回归解码阶段，每个token生成需要执行大量矩阵-向量乘法（GEMV），这本质上是内存受限的。处理-存储一体化（PIM）架构通过在HBM3 DRAM存储体内部直接执行乘累加（MAC）操作，旨在消除传统加速器中的数据搬运开销。然而，PIM的调度效率受限于DRAM时序参数。
- **识别出的关键瓶颈**：先前工作（如AttAcc、NeuPIMs）认为功率限制时序参数nCCDAB是主要性能瓶颈，并围绕它进行优化。但本文通过分析发现，对于占解码延迟90%以上的GEMV操作，DRAM行周期时间nRC比nCCDAB大10–11倍（nRC=60–77 tCK，nCCDAB=6–7 tCK），因此nCCDAB完全被nRC掩盖。即使将nCCDAB降为0，GEMV延迟也不会改变。
- **根因**：继承自主机侧的地址交错策略，使得每个全存储体MAC命令（MAC AB）在单个存储体上的地址步长为64列（2048字节），而DRAM行大小仅为32列（1024字节），导致每次MAC操作必然落入不同行，行命中率为0%，每次MAC都需要执行完整的激活-预充电（ACT-PRE）周期，开销巨大。
- **整体含义**：揭示了PIM架构下LLM推理的真正瓶颈是DRAM行管理开销，而不是先前认为的功率约束。通过简单的地址步长修改即可实现数量级的性能提升。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- RH+调度通过改变MAC AB命令的地址步长，使连续多个MAC操作访问同一DRAM行，从而将昂贵的行激活-预充电成本分摊到多个操作上，大幅减少nRC开销。

### 关键技术细节
- **步长修改**：将MAC AB地址步长从默认的64列（2048字节）改为1列（32字节）。由于每行有32列，步长为1时，连续32个MAC AB命令将依次访问同一行内的32个不同列，实现32次行命中后才需要下一个激活-预充电周期。
- **命令序列变化**：
  - 基线：每个MAC AB前必须执行ACT（激活），然后MAC（计算），最后PRE（预充电），耗时 = nRC（如63 tCK）。
  - RH+：仅第一个MAC前执行一次ACT，随后31个MAC以nCCDAB间隔连续执行，最后执行一次PRE，总耗时 = nRC + 31 × nCCDAB（例如PC模式249 tCK，对比基线2016 tCK）。
- **实现方式**：需要一次性的离线权重重排，将沿K维度的连续元素映射到同一行内的相邻列。这不改变GEMV计算结果（累加顺序可交换），且无运行时开销。
- **兼容性**：RH+不影响其他PIM命令（如WR GB、MV SB等），仅修改MAC AB的寻址模式。
