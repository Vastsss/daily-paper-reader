## 摘要
大型语言模型（LLMs）越来越多地被用于自动化电力系统分析，但许多公用事业公司和能源研究实验室出于保密、监管、可重复性和成本原因，需要本地部署。这使得开放权重模型的可靠性成为一个部署问题。我们展示了电力系统代码生成中的首次失败并非仅由推理主导，而是由结构化的API知识边界错误主导：虚构的函数名称、误用的参数以及版本化仿真库中错误处理的结果表。我们提出了PowerCodeBench，一个经过执行验证的基准生成器，将自然语言操作员查询与pandapower代码和数值真实值配对；一个L0-L3文档驱动的探测程序，用于测量每个模型的API知识概况；以及一个边界感知干预，结合查询侧API需求估计与目标主动文档注入和路由被动校正。在一个包含2000个任务的冻结版本上，我们评估了十个开放权重LLMs（1.5B-480B参数）和四个商业中端API。该干预措施改善了每个评估的至少7B参数的开放权重模型和每个商业API，准确率提升了32到56个百分点。70B-120B范围内的开放权重模型达到了商业中端准确率范围，而Llama-3.1-405B和Qwen3-Coder-480B领先。目标提示保持了全上下文准确率上限，同时只使用了41%的提示令牌成本。结果是为电网分析工作流程提供了一条无需微调或云推理的、面向准确性和部署时间的可靠本地LLM辅助路径。

## Abstract
Large language models (LLMs) are increasingly used to automate power-system analysis, but many utilities and energy-research labs require on-premise serving for confidentiality, regulatory, reproducibility, and cost reasons. This makes the reliability of open-weight models a deployment issue. We show that first-pass failures in power-system code generation are dominated not by reasoning alone, but by structured API-knowledge boundary errors: hallucinated function names, misused parameters, and mishandled result tables in versioned simulation libraries.   We introduce PowerCodeBench, an execution-validated benchmark generator that pairs natural-language operator queries with pandapower code and numerical ground truth; an L0-L3 documentation-driven probing procedure that measures per-model API knowledge profiles; and a boundary-aware intervention that combines query-side API demand estimation with targeted proactive documentation injection and routed reactive correction.   On a 2,000-task frozen release, we evaluate ten open-weight LLMs (1.5B-480B parameters) and four commercial mid-tier APIs. The intervention improves every evaluated open-weight model of at least 7B parameters and every commercial API by 32 to 56 accuracy points. Open-weight models in the 70B-120B range match the commercial mid-tier accuracy range, while Llama-3.1-405B and Qwen3-Coder-480B lead the panel. The targeted prompts preserve the full-context accuracy ceiling while using 41% of the prompt-token cost. The result is an accuracy-side, deployment-time path toward reliable on-premise LLM assistance for grid-analysis workflows without fine-tuning or cloud inference.

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

论文的核心方法是**边界感知干预框架**，由三个组件构成：

### 2.1 PowerCodeBench 基准生成器
- 一个可配置的参数化生成器，产生 `（自然语言查询，参考代码，标量真值）` 三元组。
- **设计原则**：有状态的库驱动仿真任务、可控的难度轴（单步/多步、显式参数/语义推导）、基于执行得到的数值标量真值。
- **难度分类**：D1（基础显式）、D2（多步显式）、D3（单步语义）、D4（多步语义），构成 2×2 设计。
- 冻结版本包含 2000 个任务，覆盖 15 个任务家族和 39 个电网网络。

### 2.2 L0–L3 文档驱动的知识边界探测
- 将 API 规范转化为层次化探测，测试模型对库的**识别（L0）**、**回忆（L1）**、**理解（L2）** 和**应用（L3）** 能力。
- 对每个模型 M 和每个 API 函数 f，计算风险得分：
  \[
  \rho_M(f) = 1 - \frac{1}{|\{l: s_M^{(l)}(f) \text{ defined}\}|} \sum_{l \in \{0,1,2,3\}} s_M^{(l)}(f)
  \]
  其中 \( s_M^{(l)}(f) \in [0,1] \) 是第 l 层的探测得分。高 ρ 值表示模型倾向于误用该函数。
- 探测结果揭示了跨模型一致的瓶颈模式：L2（理解）普遍饱和（>0.9），但 L0（识别）和 L3（应用）是所有模型的短板。

### 2.3 需求引导的边界干预（两阶段）

**Phase 1：主动注入（首轮生成前）**
- 输入：查询 q、模型知识概况 ρ_M、API 文档片段。
- 输出：选定的一组函数-层偶对，被注入到提示中。
- **注入分数**：
  \[
  I_\ell(f, q, M) = \hat{d}(f, q) \cdot \max(r_{M,\ell}(f), \tau_\ell(f,q)) \cdot a_\ell(q) \cdot w_\ell
  \]
  - \(\hat{d}(f,q)\)：查询端需求估计（基于混合 TF-IDF 的排名器，经角色频率校正）。
  - \(r_{M,\ell}(f)\)：模型在第 ℓ 层的缺陷（归一化）。
  - \(\tau_\ell(f,q)\)：确定性安全地板（高置信情况）。
  - \(a_\ell(q)\)：任务需要深度乘数（根据查询中的流程线索递增）。
  - \(w_\ell\)：层权重（默认均匀 0.25）。
- 采用基于令牌感知的贪婪选择来填充固定令牌预算。

**Phase 2：被动修正（运行失败后）**
- 根据执行反馈进行路由：
  - **代码错误路由**：将异常分类为 basic_fix、api_doc、boundary_contract，注入相应文档。
  - **值错误路由**（新增）：处理“代码运行但输出错误”的情况，通过 AST 追踪提取函数-输出契约（`net.res_*` 列名）和输出格式修复指令。

- 该方法的三个关键属性：模型特异性（同一查询对不同模型不同注入）、层次针对性（只注入所需层）、无需修改权重。

## 8. 不足与局限

1. **库覆盖有限**：仅基于 pandapower 库；扩展到 MATPOWER、OpenDSS 等其他后端需要重新生成 API 规范和基准。（论文承认并列为未来工作）
2. **任务格式局限**：基准仅包含产生标量结果的代码生成任务，不包括完整报告生成、交互式调度或图形输出等。
3. **应用层探测粒度较粗**：探测主要集中在函数级别，未深入参数值级别；更细粒度的参数级探测可能进一步提升 L2 干预效果。
4. **风险分数权重未优化**：L0–L3 默认均匀加权，虽然作者证明其对权重选择不敏感（1.3pp 变化），但理论上可进一步优化。
5. **转导适应假设**：需求估计的角色频率校正需要访问未标记的目标查询分布（尽管无需函数标签），在无法预采样目标场景时可能受限。作者提供了未校正基线以透明化这一影响。
6. **最低模型容量门槛**：对 1.5B 模型（Qwen2.5-Coder-1.5B）干预效果有限（峰值 <10%），方法主要适用于 7B 及以上模型。
7. **硬件指标缺失**：论文仅报告准确性和令牌效率，未提供推理延迟、总成本、能耗等实际部署指标。这些因素对于实际系统选择至关重要，但被明确定义为未来工作。
8. **商业 API 推理模式覆盖不完整**：仅测试了两个模型的推理模式（GPT-5.4-mini-high 和 DeepSeek-Reasoner），且只使用 200 项子集；其他高级推理功能未充分评估。
