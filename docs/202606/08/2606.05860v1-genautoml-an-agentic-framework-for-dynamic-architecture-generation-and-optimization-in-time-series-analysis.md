## 摘要
为时间序列预测和异常检测设计神经架构仍然是一项资源密集型任务，通常需要大量的领域专业知识。传统的自动化机器学习（AutoML）系统通常依赖于静态、预定义的搜索空间，限制了其适应多样数据特征的能力。我们提出了GenAutoML，一种利用大型语言模型（LLMs）作为神经架构师的智能体框架，用于桥接自然语言需求与可执行的PyTorch实现。该框架包含一个沙盒反思循环（Sandboxed Reflection Loop）用于自主代码优化，以及一个签名感知运行时（Signature-Aware Runtime）以确保架构一致性和执行安全。为了提高在非平稳条件下的鲁棒性，我们进一步引入了动态可逆实例归一化（Dyn-RevIN）包装器。在ETTh1、ETTm1和Weather基准上的实验表明，GenAutoML能够根据数据集特征动态生成特定任务的神经架构。在生成的模型中，WaveInterferenceNet在保持竞争性预测性能的同时，每样本推理延迟低于0.01毫秒。通过强调计算效率、架构适应性和稳定优化行为，GenAutoML能够创建适用于资源受限和延迟敏感的边缘AI部署的超轻量级神经网络。

## Abstract
Designing neural architectures for time-series forecasting and anomaly detection remains a resource-intensive task that often requires substantial domain expertise. Traditional Automated Machine Learning (AutoML) systems typically rely on static, predefined search spaces, limiting their ability to adapt to diverse data characteristics. We present GenAutoML, an agentic framework that leverages Large Language Models (LLMs) as neural architects to bridge natural-language requirements and executable PyTorch implementations. The framework incorporates a Sandboxed Reflection Loop for autonomous code refinement and a Signature-Aware Runtime that enforces architectural consistency and execution safety. To improve robustness under non-stationary conditions, we further introduce a Dynamic Reversible Instance Normalization (Dyn-RevIN) wrapper. Experiments on the ETTh1, ETTm1, and Weather benchmarks demonstrate that GenAutoML can dynamically generate task-specific neural architectures tailored to dataset characteristics. Among the generated models, WaveInterferenceNet achieves inference latency below 0.01 ms per sample while maintaining competitive predictive performance. By emphasizing computational efficiency, architectural adaptability, and stable optimization behavior, GenAutoML enables the creation of ultra-lightweight neural networks suitable for resource-constrained and latency-sensitive Edge AI deployments.

## 2. 论文提出的方法论：核心思想、关键技术细节

### 2.1 核心思想
- 将神经架构设计视为**语义到代码的映射问题**：LLM理解自然语言约束（如“捕获多粒度的季节模式”），直接生成PyTorch的`nn.Module`类。
- 采用**闭环自适应系统**：生成→沙盒测试→错误反馈→再生成，迭代直至架构有效。

### 2.2 关键技术细节

#### （1）**沙盒反思循环（Sandboxed Reflection Loop）**
- 将LLM生成的模型`Mgen`放入隔离沙盒，用模拟张量`X_dummy`执行前向传播。
- 程序化断言输出形状是否符合几何约束：`X(B,L,D) → Y(B,H,D)`。
- 若失败，捕获Python回溯和维度错误，反馈给LLM作为额外提示，迭代修正。实验表明所有架构最终100%通过验证（最多5次迭代）。

#### （2）**签名感知运行时与形状无关投影（Signature-Aware Runtime & Shape-Agnostic Projection）**
- **签名感知调用**：利用Python `inspect.signature`自动匹配运行时张量到生成模块的定位参数，避免因前向方法签名未知导致崩溃。
- **形状无关投影层**：若生成的隐藏通道数与预测目标维度不匹配，自动插入线性投影`Ŷ = W_proj · Mgen(Ẋ) + b_proj`，将任意尺寸的潜在特征映射到目标维度，无需LLM一次写出完美输出层。

#### （3）**动态可逆实例归一化（Dyn-RevIN）**
- 采用RevIN框架的数学公式，作为**统计护栏**，解耦不可信的生成架构逻辑与严格平稳性约束。
- 流程：对输入`X`计算实例均值和方差 → 归一化为平稳分布`Ẋ` → 通过生成模型`Mgen`得到`Ŷ` → 反归一化恢复原始统计域。
- 公式：
  - 归一化：`Ẋ_{i,k,j} = (X_{i,k,j} - μ_{i,j}) / sqrt(σ²_{i,j} + ε)`
  - 模型映射：`Ŷ = Mgen(Ẋ)`
  - 反归一化：`Y_{i,k,j} = Ŷ_{i,k,j} · sqrt(σ²_{i,j} + ε) + μ_{i,j}`

#### （4）**端到端对话管线**
- 用户上传CSV/Parquet数据集 → LangChain + Pandas DataFrame Agent执行探索性数据分析（EDA） → 用户要求设计自定义架构 → 沙盒验证 → 通过**JIT动态加载器**将新类直接注入**Optuna**优化循环（热交换，无需重启）。

### 2.3 算法/流程概述
1. 用户与对话式数据代理交互，进行EDA。
2. 用户用自然语言描述架构需求。
3. LLM生成PyTorch代码，提交至沙盒反思循环。
4. 沙盒用模拟张量测试，反馈错误给LLM修正。
5. 通过后代码保存至动态注册表。
6. JIT加载器解析AST并注入Optuna搜索空间（与基线并列）。
7. 优化循环使用早停、衰减学习率等进行超参数调优，输出最终模型。

## 6. 论文的主要结论与发现

1. **动态生成架构具有竞争力**：生成的ResNet、Inception等模型在预测任务中接近复杂Transformer（如TimesNet）的性能，验证了LLM能自主发现高效的归纳偏置。

3. **超轻量、极低延迟**：WaveInterferenceNet（82.9万参数）推理延迟<0.01ms，比Chronos-T5-Mini快约10万倍（0.01ms vs 987.93ms）。

5. **沙盒反思循环保证可靠性**：所有生成架构经闭环迭代后100%通过验证，复杂模型最多需5次迭代。
6. **确定性优势**：WaveInterferenceNet在3个种子下MAE标准差为0（零方差），而DLinear的标准差高达0.76，但平均MAE略高（2.1485 vs 0.8780），说明精度与确定性之间存在权衡。

## 7. 优点

- **创新性**：率先将LLM作为真正的“架构工程师”，实现从自然语言需求到可执行代码的动态生成与闭环优化，突破了传统NAS静态搜索空间的限制。
- **工程实用性**：JIT热交换注入、签名感知调用、形状无关投影等技术解决了生成代码与训练管线的集成难题，具备工业部署潜力。
- **边缘AI导向**：明确以微秒级延迟和确定性收敛为目标，区别于追求极致精度但代价高昂的基础模型。
- **统计护栏设计**：Dyn-RevIN作为可选的数学保障层，通过消融揭示了其双面性，体现了严谨性。
- **可解释性与可追溯性**：提供反思日志和对话式界面，便于用户理解和调试生成过程。

## 8. 不足与局限

- **LLM依赖性**：当前依赖云端LLM API，每次生成需30-60秒，在时间敏感场景下可能是瓶颈。未来需探索本地小模型优化。
- **实验覆盖有限**：仅评估三个公开数据集（电力、天气），未涵盖金融、交通、医疗等更多领域；也未在更多基础模型（如MOIRAI）上对比。
- **稳定性验证不充分**：仅对WaveInterferenceNet做了3种子的稳定性测试，且其平均MAE远高于DLinear，说明确定性以精度为代价。其他生成模型的方差未知。
- **与NAS方法的直接对比缺失**：未与LLMatic、NADER等基于LLM的NAS方法在相同搜索预算下对比。
- **任务局限**：仅关注数值时间序列，不支持多模态（文本、图像、日志）数据融合。
- **搜索成本**：尽管Optuna微搜索成本低，但LLM调用成本（API费用、时间）仍存在，且未与端到端NAS的GPU小时数详细对比。
- **无理论保证**：框架依赖LLM的生成能力，缺乏对生成架构性能的数学上界或是率一致性证明。
