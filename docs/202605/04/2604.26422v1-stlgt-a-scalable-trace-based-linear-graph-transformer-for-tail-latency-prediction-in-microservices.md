## 摘要
准确的端到端尾部延迟预测对于微服务系统中的主动 SLO 管理至关重要。然而，在保持大规模推理效率的同时，对长程依赖传播以及非平稳、突发性工作负载进行建模仍然具有挑战性。我们提出了 STLGT（可扩展的基于追踪的线性图 Transformer），这是一种针对每个 API 的预测器，它将追踪编码为跨度图（span graphs），用于多步 p95 尾部延迟预测。STLGT 使用结构感知的线性图 Transformer 来传播跨服务依赖关系，其推理时间与跨度图的大小呈线性关系，并使用解耦的时间模块来捕获工作负载动态。在个性化教育微服务应用、DeathStarBench 和阿里巴巴追踪数据集上的实验表明，STLGT 的预测准确率比 PERT-GNN 平均提高了 8.5% 的 MAPE，并且在 N=32（匹配阿里巴巴追踪预处理后的最大跨度图大小）时，CPU 推理速度提高了多达 12 倍。消融研究进一步证明了各组件的有效性，特别是在突发流量下。

## Abstract
Accurate end-to-end tail-latency forecasting is critical for proactive SLO management in microservice systems. However, modeling long-range dependency propagation and non-stationary, bursty workloads while maintaining inference efficiency at scale remains challenging. We present STLGT (Scalable Trace-based Linear Graph Transformer), a per-API predictor that encodes traces as span graphs for multi-step p95 tail-latency forecasting. STLGT uses a structure-aware linear graph Transformer to propagate cross-service dependencies with inference time linear in span graph size, and a decoupled temporal module to capture workload dynamics. Across a personalized education microservice application, DeathStarBench, and Alibaba traces, STLGT improves forecasting accuracy over PERT-GNN by 8.5% MAPE on average and achieves up to 12x faster CPU inference at N=32, matching the maximum span graph size after preprocessing the Alibaba traces. Ablation studies further demonstrate the effectiveness of each component, especially under bursty traffic.

### 2. 方法论
STLGT 采用了一种**基于追踪（Trace）的线性图 Transformer** 架构，核心步骤如下：
*   **Span 图构建**：从分布式追踪（Distributed Tracing）中提取 Span 层次结构，构建 API 特有的 Span 图。节点代表执行阶段（(调用者, 被调用者) 对），边代表调用关系。
*   **空间编码器（Spatial Encoder）**：
    *   **结构感知线性注意力**：通过线性化注意力机制将复杂度从 $O(N^2)$ 降至 $O(N)$。在计算注意力前，利用图拓扑结构对 Key 和 Value 进行“一次性稀疏预混合”，引入结构先验。
    *   **局部传播分支**：并行使用一个轻量级的 GCN 层来捕捉局部邻域的依赖关系。
*   **时间解码器（Temporal Decoder）**：
    *   **解耦建模**：将空间依赖编码与时间动态建模分离。
    *   **TimesNet 模块**：利用快速傅里叶变换（FFT）识别多周期模式，通过 TimesBlock 捕捉周期性波动和非周期性突发。
*   **损失函数**：使用分位数回归损失（Pinball Loss），专门针对 p95 尾部延迟进行优化。

### 7. 优点
*   **线性复杂度**：成功将 Transformer 的全局建模能力引入微服务领域，同时解决了 $O(N^2)$ 的计算瓶颈。
*   **结构与特征融合**：通过 Span 图不仅捕捉了服务间的拓扑，还融合了细粒度的追踪指标（如 Span 持续时间）和粗粒度的服务指标（如 CPU 利用率）。

### 8. 不足与局限
*   **拓扑漂移风险**：模型假设在评估期内 API 的主调用路径是稳定的。如果微服务架构发生频繁的动态路由变更或版本更新，可能需要频繁重构 Span 图并重新训练。
